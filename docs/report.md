# Отчёт по варианту 9

Дополнительный набор схем проекта вынесен в [diagrams.md](diagrams.md).

## 1. Тема

Разработка системы интеллектуального управления пропускной способностью спутникового канала связи на основе прогнозирования трафика.

## 2. Актуальность

Спутниковый канал связи обладает ограниченной пропускной способностью и работает в условиях изменяющейся нагрузки. Если ресурс выделяется статически, это приводит либо к перегрузкам в моменты пикового трафика, либо к избыточному резервированию в периоды спада нагрузки. Поэтому практически значимой задачей является разработка системы, которая заранее оценивает будущую загрузку канала и адаптирует выделяемую полосу пропускания.

## 3. Цель и задачи

Цель работы состоит в разработке имитационной системы управления пропускной способностью спутникового канала на основе прогноза временного ряда нагрузки.

Для достижения цели были решены следующие задачи:

1. Выбрать открытый датасет с временными рядами сетевой нагрузки.
2. Сформировать агрегированный почасовой ряд, интерпретируемый как нагрузка спутникового канала.
3. Построить признаки для прогнозирования без использования будущей информации.
4. Сравнить несколько моделей прогноза.
5. Разработать правило адаптивного выделения пропускной способности.
6. Сравнить adaptive policy с baseline-подходом фиксированной ёмкости.

## 4. Используемые данные

В работе использован открытый датасет CESNET-TimeSeries24 в виде sample-архива из каталога [datasets](../datasets). Данный набор содержит агрегированные временные ряды сетевой активности по нескольким масштабам времени. Для проекта выбрана агрегация `1 hour`.

Важно, что используемый набор **не является прямым спутниковым измерением**. Он применяется как открытый источник временного ряда нагрузки, на основе которого строится имитационная модель спутникового канала связи.

В качестве нагрузки канала использована суммарная величина `n_bytes` по пяти наиболее активным и достаточно полным рядам:

- `11069`
- `103`
- `1368`
- `10256`
- `1367`

После формирования признаков итоговый размер выборки составил `6693` наблюдения.

## 5. Метод решения

Метод решения включает два связанных блока:

1. Прогнозирование будущей нагрузки канала.
2. Принятие решения о выделении пропускной способности по результатам прогноза.

Для подготовки данных были выполнены:

- восстановление временной оси через `times_1_hour.csv`;
- построение календарных признаков: `hour`, `day_of_week`, `month`, `is_weekend`, `is_holiday`;
- построение циклических признаков `sin/cos`;
- формирование лагов `1, 2, 3, 6, 12, 24, 48, 72`;
- расчёт скользящих средних и стандартных отклонений.

Для исключения leakage все rolling-признаки и производные показатели строились только по прошлым значениям, со сдвигом минимум на один шаг назад.

Сравнивались три модели:

- `NaiveLastValue`;
- `SeasonalNaive24`;
- `OLSRegression`.

Далее лучшая модель использовалась в модуле управления ресурсом. Для сравнения были заданы две политики:

- `BaselineFixedCapacity` — фиксированная выделяемая ёмкость;
- `ForecastDrivenCapacity` — адаптивная ёмкость на основе прогноза и резервного запаса.

## 6. Архитектура системы

Разработанная система включает четыре блока:

1. Блок сбора и подготовки телеметрии.
2. Блок прогнозирования нагрузки.
3. Блок управления пропускной способностью.
4. Блок оценки качества решения.

Логика работы системы следующая:

1. Из датасета формируется агрегированный ряд нагрузки канала.
2. Прогнозный модуль оценивает ожидаемую нагрузку на следующий интервал.
3. Модуль управления рассчитывает требуемую выделяемую ёмкость.
4. На тестовом интервале сравнивается результат adaptive policy и baseline-подхода.

Таким образом, интеллектуальность системы определяется не только наличием прогноза, но и использованием прогноза как входа для управляющего решения.

## 7. Результаты эксперимента

Результаты прогноза приведены в [model_metrics.csv](../results/model_metrics.csv).

| Модель | MAE | RMSE | MAPE, % | R2 |
| --- | ---: | ---: | ---: | ---: |
| NaiveLastValue | 1.56e9 | 2.20e9 | 20.90 | 0.309 |
| SeasonalNaive24 | 2.27e9 | 3.03e9 | 32.66 | -0.304 |
| OLSRegression | 3.12e9 | 1.39e10 | 38.25 | -26.389 |

Лучшей моделью на выбранных данных оказалась `NaiveLastValue`. Такой результат интерпретируется как следствие высокой инерционности временного ряда.

## 8. Сравнение baseline и adaptive policy

Результаты сравнения политик приведены в [policy_metrics.csv](../results/policy_metrics.csv).

| Политика | Overload rate, % | Served traffic ratio, % | Total overload bytes |
| --- | ---: | ---: | ---: |
| BaselineFixedCapacity | 2.91 | 99.23 | 8.28e10 |
| ForecastDrivenCapacity | 0.67 | 99.90 | 1.05e10 |

Ключевые эффекты адаптивного управления:

- доля перегрузок снизилась с `2.91%` до `0.67%`;
- доля обслуженного трафика выросла с `99.23%` до `99.90%`;
- суммарная перегрузка сократилась примерно в `7.86` раза.

## 9. Вывод

В работе разработана имитационная система интеллектуального управления пропускной способностью спутникового канала на основе прогноза трафика. Система объединяет прогнозный модуль и модуль принятия решения по выделению ресурса.

Полученные результаты показывают, что даже при использовании простой и честно верифицированной модели прогноза возможно существенно снизить перегрузки канала и повысить долю обслуженного трафика.

## 10. Ограничения работы

- использован не спутниковый измерительный датасет, а открытый сетевой временной ряд;
- управление реализовано в виде rule-based policy;
- не учитывались физические параметры канала;
- глубина модельной части ограничена учебным форматом проекта.
